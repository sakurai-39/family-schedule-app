## 各決定事項の詳細と背景

### #1. 対応OS：iOS + Android両対応（必須）

- 夫：Pixel 6a（Android）
- 妻：iPhone X（iOS）
- → React Native (Expo) で両対応必須

**注意点**：
- iPhone XはiOS 16まで対応・問題なく動く
- App Store公開時はApple Sign-In実装必須（Apple審査要件）

### #2. バックエンド：Firebase

**選定理由**：
1. プッシュ通知（FCM）が両OS対応で実績多数
2. 日本語チュートリアル豊富・Codex/Claude Codeが書きやすい
3. 夫婦+子ども最大4ユーザーで無料枠完全に収まる
4. Pixel ユーザーならGoogle Sign-Inが1タップ

**料金リスク対策（実装計画書に組み込む必須項目）**：
1. Sparkプランで開発・初期リリース（クレカ登録しない）
2. セキュリティルールはMVPの最初から厳格に書く
3. 読み取り回数を最小化する設計（リアルタイム監視は必要箇所のみ）
4. App Check導入（フェーズ2で実装）
5. 本番リリース前に予算アラート設定手順を文書化

### #3. 認証：Google + Apple Sign-In

- 妻はFace ID/Touch IDで1タップログインに慣れている
- パスワード入力ゼロ＝挫折ポイントなし
- App Store審査要件（他ソーシャルログインを使う場合Apple Sign-Inが必須）
- Firebase Authで両方とも標準サポート

### #4. ペアリング：招待リンク + 6桁コード

**フロー**：
1. Ryouさんがアプリインストール → サインイン → 「家族を作成」
2. 招待リンク（`familycal://invite/abc123`）と6桁コード（`482917`）が表示される
3. RyouさんがLINEで奥さんに招待リンク送信
4. 奥さん：iPhoneでリンクタップ → アプリ起動 → サインイン → 自動で紐付き完了
5. リンクが効かない場合のバックアップとして6桁コード入力でも紐付け可能

**実装**：Firebase Dynamic Links機能を利用

**家族メンバー上限**：MVPは2人（夫婦）。将来子ども追加できるよう「household（家族）に複数メンバー所属可能」な構造で設計。

### #5. データ構造：C案（カレンダー+買い物分離）

```
households コレクション
  └─ {householdId}
       ├─ members: [userId1, userId2]
       ├─ calendar_items サブコレクション
       │    └─ {itemId}
       │         ├─ type: 'event' | 'task'
       │         ├─ title: string
       │         ├─ assignee: 'papa' | 'mama' | 'both' | 'whoever'
       │         ├─ startAt / dueAt: timestamp
       │         ├─ isCompleted: boolean
       │         ├─ recurrence: null  ← MVPでは常にnull、将来用に予約
       │         └─ notifySettings: { ... }
       └─ shopping_items サブコレクション
            └─ {itemId}
                 ├─ name: string
                 ├─ category: 'consumable' | 'school' | 'other'
                 ├─ isStockItem: boolean
                 └─ isCompleted: boolean
```

**選定理由**：
- 予定とタスクは「カレンダー画面で一緒に表示する必要がある」ため統合
- 買い物リストは「いつ買うか」より「何を買うか」が主役で別画面 → 分離
- カレンダー画面のクエリが1回で済むため料金最適化

### #6. 繰り返し予定：MVPは単発+複製ボタン

**MVPで実装**：
- 単発の予定の追加・編集・削除
- 複製機能：
  - 翌日にコピー
  - 翌週同じ曜日にコピー
  - 翌月同じ日にコピー
  - 期間を指定して一括コピー（例：来月いっぱい毎週水曜）

**選定理由**：
- 「動くMVPを早くリリース」して妻が使ってくれるか検証することが最優先
- 繰り返し機能は例外処理（特定日だけ削除/時刻変更等）が大量にあり、初心者プロジェクトには重い
- シフト勤務は不定期なので「繰り返し」が必須ではない
- 習い事・ゴミ出しは「期間指定一括コピー」で代替可能

**将来の拡張性**：`recurrence`フィールドを最初から予約。Phase 2で値を入れて機能追加可能。

### #7. 通知仕様

**①タイミング**（固定値）：
- 朝8時にまとめて通知（その日の予定一覧）
- 各予定の1時間前にリマインド

**②受信者ルール**：
- 担当者「papa」 → Ryouさんのみ通知
- 担当者「mama」 → 妻のみ通知
- 担当者「both」 → 夫婦両方に通知
- 担当者「whoever」 → 夫婦両方に通知（その時できる方が対応）

**③タスク完了の相手通知**：
- 完了時に相手に即時プッシュ通知
- 例：「ママが連絡帳記入を完了しました」
- フェーズ1ではローカル通知のみ（Expo Goの制約）、フェーズ2でプッシュ通知本格実装

**MVPの設定画面**：なし。固定値で動作。将来の要望次第でPhase 2に設定画面追加。

### #8. オフライン：完全対応

**実装**：Firestoreの`enablePersistence()`で標準サポート（ほぼ実装コストゼロ）

**競合解決ルール**（実装計画書に明記）：
| ケース | 解決方法 |
|---|---|
| 夫婦が別々の予定を圏外で追加 | 両方とも同期。問題なし |
| 夫婦が同じ予定を圏外で編集 | 後勝ち（最後に同期した方の内容を採用） |
| 同期失敗・サーバーエラー | アプリ内に「未同期」バッジを表示 |

**プッシュ通知のオフライン対応**：朝8時/1時間前は端末のローカル通知として予約 → Firebase繋がらなくても通知が飛ぶ。

### #9. iOS配布戦略

**段階的移行**：
```
フェーズ1（MVP）：B（Expo Go） — 0円
   ↓ 1〜2ヶ月使ってみて確信を得たら
フェーズ2：B継続 + 必要ならC（Development Build + TestFlight）に移行
   ↓ 「絶対公開したい」と決めた時
フェーズ4：A（App Store公開）— $99/年支払い
```

**フェーズ1でB（Expo Go）を選ぶ理由**：
- 「妻に使ってもらえるか」の検証が最優先
- Geminiの「コスト0円」方針と整合
- MVP機能（カレンダー・予定追加・ペアリング）はすべて動く
- 使われなかった場合$99を捨てずに済む
- 使えると確信した時点でAに移行可能（データはFirebaseに保存されているのでアプリだけ作り直せばOK）

**注意**：プッシュ通知はExpo Goで制限あり。フェーズ2でCに移行する時に有効化する形。
